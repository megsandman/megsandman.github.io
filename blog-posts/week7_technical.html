Tech Talk: SQL v. NoSQL
September 20, 2014

This week we dove into SQL - short for Structured Query Language - and have had an introduction to databases and database management systems (DMS). The DMS is the piece that handles the sets of information and SQL is a language that gives users a way to interact with that data. SQL allows you to create a database, add or delete information from it, or pull information from it based on specified parameters.

SQL databases are relational databases. This means that the data is stored in tables (also known as relations) with each table contains rows, which each contain a unique record, and columns, which are also referred to as attributes. You can create relationships between the tables to form a very powerful and large dataset that is easier to access and query than if all the data had been stored in one single table.

Some of the benefits of SQL include:

  • Ease of use. Even after just few days of learning SQL I've been surprised by how much I'm able to do. The syntax is very readable and easy to pick up and with just a small set of commands users are able to perform a powerful number of operations.
  • Standardization. SQL is an established standard that has been used for a long time so there is agreement among the community on best practices and a large amount of open source work available.
  • Complex data. SQL databases handle complex data very well and allows users to easily and quickly perform powerful operations.

And a few of the drawbacks:

  • It's rigid. When creating a database, you must use a predefined schema, which means that you must know the structure of the data set you will be collecting ahead of time. This can be too rigid in fields like web development.
  • SQL only works in tables. This is not always the best way to store some data types, like hierarchical data.
  • Scale vertically. SQL databases are easy to scale vertically but not horizontally, which means that in order to increase the power of the database you need to increase the load on each node rather than spreading the load out over additional nodes.

Because of some of the problems mentioned above with SQL several other languages have emerged. One of those is NoSQL, which is short for Not Only SQL. While SQL databases are relational, NoSQL databases are non-relational (also called distributed), which means that instead of being stored in tables the data is stored in key-value pairs, graphs, documents, or columns.

Some of the benefits of NoSQL:

  • Less structured than SQL. NoSQL databases are dynamic and do not require a schema to be pre-defined.
  • Scale horizontally. You can increase the load of the database by adding additional nodes and reducing the load on each node.
  • Good with large data sets and hierarchical data sets because NoSQL databases allow for data stored in key-value pairs.

There are also a few drawbacks of NoSQL:

  • Less standardization. Because it is much newer and less rigid, NoSQL does not have the standards and level of best practices in place that SQL has. This also makes it more risky for use of sensitive data.
  • Difficulty with complex queries. SQL is more adept at handling complex queries than SQL.

As I continue to learn more about both languages, I'm excited to better understand the ins and outs of each so that I'll be able to identify the best database structure for projects I work on.
